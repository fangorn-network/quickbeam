  ## 🏹 5. Tactical On-Chain Playbooks & Diagnostics

  ### 🟩 Practical On-Chain Playbooks

  *   **Play 1 — Mask the plumbing before ranking anything.** Exclude the twelve PT ≥ 0.998 conduits — plus C2 `0x1A18a8b9` at 0.9979 — from every volume leaderboard. The only address whose accumulation survives that correction is **`0x9c856e12...1a65D664`** ( + $401,160 in, $0 out, 24 tx ), followed by `0x2F4579Ca` ( + $116,266, PT 0.594 ). **Those two are the real positioning signal on this chain.** Watch them.

Either recalibrate it to spread across [0,1] using the components that *do* discriminate ( `circularityRatio` , `amountQuantization` , `senderHHI` ), or drop it so consumers stop ranking on noise.
  5.  **Emit an `actionType` vocabulary with actual variance, and label known addresses.** All 17,837 transfers are `actionType = "transfer"` / `signal = "notable-transfer"` — mint and burn are currently only inferable by string-matching the zero address. Classify `mint` / `burn` / `conduit-hop` at ingest, and ship a canonical tag table for the pipeline addresses ( C1 `0xcfAEce21` = mint relay, `0xa8553DB0` = burn router ) so every downstream consumer does not have to re-derive the topology from scratch, as this report did.
  6.  **Add an `entityType = "Wallet"` metrics payload.** All 2,369 wallet records carry only `address` + `text` , so every entity statistic in §3 — pass-through, breadth, net USD, hop latency — had to be recomputed from raw edges. The asset-level fields ( `senderHHI` , `circularityRatio` ) prove the pipeline already knows how to do this; extend it to wallets and expose `passThroughRatio` directly.
